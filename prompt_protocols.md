## 2. Tutor Implementation (D2)

**Method**: MathDial human tutor turn-by-turn replay.

Each original MathDial tutor turn is passed verbatim as the `user` message
to the learner LLM. Turn budget = `n_tutor_turns_original + 1` (for transfer probe).

All four conditions (C1–C4) receive identical tutor stimuli for the same scenario.
This eliminates tutor behaviour as a confound across conditions.

**Transfer probe**: auto-generated by replacing numbers in the original problem
(same structure, different values). Appended as the final user message in all conditions.
Only C1 protocol requires a genuine independent attempt.

## 3. Four Condition Protocols

### C1 — Teachable Agent

**Framework**: Blair et al. (2007) — independent performance, productive learner behaviour,
support for teaching, visible reasoning as shared representation.

**Required structural elements** (protocol, not wording):
1. **Ask questions**: learner must clarify when confused (ablatable: `ablate_questions`)
2. **Revise explicitly**: learner must state revision after feedback (ablatable: `ablate_revise`)
3. **Maintain student stance**: no teaching or explaining upward (ablatable: `ablate_student_stance`)
4. **Independent transfer**: learner must attempt transfer probe independently (ablatable: `ablate_transfer`)

**Prompt file**: `prompts/c1_teachable_agent.txt`  
**Success evidence**: TA1 (independent performance), TA2 (productive learner), TA3 (support for teaching), TA4 (visible reasoning)

---

### C2 — Student Simulation

**Framework**: Koedinger et al. (2015) — cognitive model plausibility, error-model alignment,
prior-knowledge consistency, learning-process plausibility, instructional-testing utility.

**Required structural elements**:
1. **First-turn misconception**: target misconception injected via `student_profile` + `teacher_described_confusion` (ablatable: `ablate_misconception`)
2. **Verbalize reasoning**: always show reasoning steps (ablatable: `ablate_verbalize`)
3. **Gradual learning**: no single-turn full correction (ablatable: `ablate_gradual`)
4. **Prior knowledge boundary**: refuse concepts above 7th grade (ablatable: `ablate_prior_knowledge`)

**Prompt file**: `prompts/c2_student_simulation.txt`  
**Success evidence**: SS1–SS5

---

### C3 — Generic Learner (Baseline)

**Prompt**: Single instruction only: "Act as a math student responding to the tutor."  
**No structural constraints.**  
**Purpose**: Tests whether effects arise from any student-role wording vs structured protocols.  
**Prompt file**: `prompts/c3_generic_learner.txt`

---

### C4 — No-role Assistant (Control)

**Prompt**: Standard helpful-assistant system prompt.  
**No learner role assigned.**  
**Purpose**: Controls for ordinary LLM answer behaviour.  
**Prompt file**: `prompts/c4_no_role_assistant.txt`

## 4. Turn Budget Formula

```
turn_budget = len(original_mathdial_tutor_turns) + 1
```

The `+1` is the transfer probe turn, identical across all conditions.

## 5. Exclusion Rules (Pre-registered)

| Code | Trigger | Action |
|------|---------|--------|
| E1 | Empty learner turn (<5 chars) | Drop turn; ≥2 such turns → exclude dialogue |
| E2 | Non-math content (off-topic, detected by keyword or LLM classifier) | Exclude dialogue |
| E3 | Unsafe / refusal response | Exclude dialogue, log refusal text |
| E4 | Token overflow (>800 tokens in single turn) | Truncate, log; if final answer lost → exclude |
| E5 | Missing required turn (transfer probe response absent in C1) | Regenerate once; if fails → exclude |
| E6 | Condition leakage (learner outputs role label from prompt) | Exclude dialogue |

All exclusions are reported by condition in the paper Appendix.
**No exclusion rule may be added after pilot generation begins.**

## 6. Protocol-Not-Wording Guarantee

C1 and C2 differ by required **interaction structure** and **success criterion**,
not only by phrase labels. Specifically:

- C1 requires near-transfer attempt at dialogue end; C2 does not.
- C2 requires injection of a specific `teacher_described_confusion` from MathDial metadata; C1 does not.
- C3 has neither structural requirement.
- These structural differences are independently ablatable (see §3).

## 7. Scenario Parity

Every scenario is generated under every condition and every seed.
If one (scenario × condition × seed) fails, only that combination is regenerated.
Reason for regeneration is logged in `outputs/{phase}/{condition}/{scenario_id}_{seed}.json` → `exclusion_reason`.
